# Инструкция для агентов

## Что это за проект

«Караси против Трески» — небольшая браузерная игра на HTML5 Canvas. В проекте нет сборщика, package manager, backend и модулей ES. Точка входа — `index.html`, игровые классы объявляются глобально через `window`.

Главная задача при изменениях: сохранить запуск игры простым открытием `index.html` или через любой статический HTTP-сервер.

## Структура

```text
index.html                         разметка, HUD, стили и порядок подключения JS
src/main.js                       загрузка ассетов и создание GameScene
src/GameScene.js                  игровой цикл, спавн, столкновения и рендер
src/Player.js                     игрок, урон, лечение и стрельба
src/Enemy.js                      противники
src/Obstacle.js                   ботинок и крюк
src/Bullet.js                     снаряды
src/HealingPickup.js              лечебный бинт
src/Effects.js                    взрыв и визуальный эффект лечения
src/Assets.js                     пути к изображениям и карта атласа
assets/sprites/source/             исходники, добавленные пользователем или генератором
assets/sprites/runtime/            очищенные и готовые к игре PNG
assets/sprites/game-sprites.png    действующий атлас основных объектов
tools/remove-sprite-chroma.ps1     удаление однотонного зелёного фона
```

## Архитектура игры

`GameScene` владеет всеми коллекциями сущностей:

- `players`;
- `bullets`;
- `enemies`;
- `obstacles`;
- `healingPickups`;
- `effects`.

Каждая сущность обычно реализует:

```js
update(delta) {}
draw(ctx, assets) {}
```

Для удаления сущности нужно выставить `active = false`. Физическое удаление из массива выполняется в `GameScene.cleanup()`.

Координаты `x` и `y` считаются центром сущности. `width` и `height` используются для прямоугольных коллизий и должны примерно соответствовать видимой части спрайта, а не размеру исходного PNG с отступами.

## Порядок игрового кадра

В `GameScene.update()`:

1. Обновляется фон.
2. Обновляются таймеры и игрок.
3. Создаются новые сущности.
4. Вызывается `update()` у сущностей.
5. Обрабатываются попадания пуль.
6. Обрабатываются столкновения игрока.
7. Удаляются неактивные сущности.
8. Обновляется HUD.

Не помещать подбор предметов в `resolveBulletHits()`. Бонусы обрабатываются только в `resolvePlayerCollisions()`.

## Подключение новых классов

ES-модули не используются. В конце файла класс экспортируется так:

```js
window.NewEntity = NewEntity;
```

Затем файл добавляется в `index.html` до `GameScene.js`. Порядок `<script>` важен: зависимость должна быть подключена раньше потребителя.

## Работа с ассетами

Новые изображения от пользователя сначала искать в `assets/sprites/source/`.

Предпочтительный формат нового ассета:

- отдельный PNG на один объект;
- прозрачный фон;
- если прозрачность невозможна — строго однотонный `#00ff00`;
- объект целиком в кадре и расположен по центру;
- без текста, рамок, теней и дополнительных предметов;
- имя файла — латиница, kebab-case, без пробелов.

Исходник не менять. Подготовленный вариант сохранять в `assets/sprites/runtime/` под понятным именем. Не подключать случайные UUID-имена напрямую из `source`.

Если у изображения зелёный фон, его можно обработать так:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass `
  -File .\tools\remove-sprite-chroma.ps1 `
  -InputPath .\assets\sprites\source\asset-chroma.png `
  -OutputPath .\assets\sprites\runtime\asset.png
```

После обработки проверить:

- наличие alpha-канала;
- прозрачность углов;
- отсутствие зелёной каймы;
- разумный crop без огромных пустых полей;
- соответствие размера отрисовки коллизиям.

Для новых одиночных ассетов предпочтительнее отдельные PNG. Не расширять общий атлас без необходимости: ручные координаты нарезки легко ломаются.

Все изображения загружаются в `src/Assets.js`. Добавлять загрузку в общий `Promise.all`, чтобы игра не стартовала до готовности ресурсов.

## Текущие параметры баланса

- здоровье игрока: `100 HP`;
- бинт лечит `25 HP` без превышения максимума;
- первый бинт появляется примерно через `7 секунд`;
- следующие бинты — через `10–16 секунд`;
- на полном здоровье бинт не расходуется;
- автоматическая стрельба игрока: каждые `0.22 секунды`.

Баланс хранить рядом с логикой сущности или спавна. Не размазывать одно значение по нескольким файлам.

## Кодировка

Все текстовые файлы должны оставаться в UTF-8. В проекте есть русские строки. PowerShell 5 может неверно отображать UTF-8 без BOM через обычный `Get-Content`; это не всегда означает, что браузер получит битый текст. Для проверки явно указывать `-Encoding UTF8` или проверять байты/страницу в браузере.

Не выполнять массовую перекодировку файлов без проверки результата: так легко заменить русские символы на `�`.

## Проверки перед завершением

Минимальная проверка синтаксиса:

```powershell
Get-ChildItem .\src\*.js | ForEach-Object { node --check $_.FullName }
```

Также вручную или автоматизированно проверить:

- все пути из `Assets.js` существуют;
- новые скрипты подключены в правильном порядке;
- состояние новой механики очищается в `resetWorld()`;
- таймер механики также сбрасывается;
- сущность обновляется, рисуется и удаляется через `cleanup()`;
- здоровье не становится больше `maxHealth` и меньше нуля;
- коллизия соответствует видимому объекту;
- старт, рестарт и game over продолжают работать.

## Критерии готовности изменения

Изменение считается законченным, когда оно подключено в реальный игровой цикл, использует runtime-ассет, корректно сбрасывается при рестарте, проходит `node --check` и проверено хотя бы одним целевым сценарием поведения. Просто добавить файл или класс недостаточно.
