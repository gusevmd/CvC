# Инструкция для агентов

## Что это за проект

«Караси против Трески» — небольшая браузерная игра на HTML5 Canvas. В проекте нет сборщика, package manager, backend и модулей ES. Точка входа — `index.html`, игровые классы объявляются глобально через `window`.

Главная задача при изменениях: сохранить запуск игры простым открытием `index.html` или через любой статический HTTP-сервер.

## Структура

```text
index.html                         разметка, HUD, стили и порядок подключения JS
src/main.js                        загрузка ассетов, выбор режима и класса, создание GameScene
src/GameScene.js                   игровой цикл, стадии кампании, спавн, столкновения, HUD и post-run shop
src/Player.js                      игрок, урон, лечение, свободное движение и стрельба
src/PlayerClasses.js               конфиги классов игрока
src/Enemy.js                       обычные враги, стрелки и босс
src/Obstacle.js                    ботинок, крюк и сундук
src/Bullet.js                      снаряды, ракеты и их визуал
src/HealingPickup.js               лечебный бинт
src/UpgradePickup.js               апгрейд на двойной выстрел
src/CoinPickup.js                  монеты и их разлёт
src/Effects.js                     взрыв, импакт и визуальный эффект лечения
src/Assets.js                      пути к изображениям и карта атласа
assets/sprites/source/             исходники, добавленные пользователем или генератором
assets/sprites/runtime/            очищенные и готовые к игре PNG
assets/sprites/game-sprites.png    действующий атлас основных объектов
tools/remove-sprite-chroma.ps1     удаление однотонного зелёного фона
```

## Текущее состояние игры

Сейчас в проекте есть два режима:

- `campaign` — короткий конечный уровень со стадиями, переходом в боссфайт и экраном победы;
- `endless` — бесконечный режим без финального босса.

На стартовом экране игрок выбирает:

- режим;
- класс карася: `Стрелок`, `Инженер`, `Медик`.

После смерти или победы открывается overlay результата с:

- итоговым счётом;
- кошельком;
- покупкой постоянных апгрейдов на текущую сессию.

Постоянные апгрейды сейчас такие:

- здоровье;
- темп огня;
- магнит монет;
- скорость.

## Архитектура игры

`GameScene` владеет всеми коллекциями сущностей:

- `players`;
- `bullets`;
- `enemyBullets`;
- `enemies`;
- `obstacles`;
- `healingPickups`;
- `upgradePickups`;
- `coinPickups`;
- `boss`;
- `effects`.

Также `GameScene` хранит состояние run-level и meta-level систем:

- `selectedMode`;
- `selectedClassId`;
- `campaignStages` и текущее состояние стадии;
- `wallet`;
- `metaUpgrades`;
- таймеры спавна;
- состояние boss encounter и post-run shop.

Каждая сущность обычно реализует:

```js
update(delta) {}
draw(ctx, assets) {}
```

Для удаления сущности нужно выставить `active = false`. Физическое удаление из массива выполняется в `GameScene.cleanup()`.

Координаты `x` и `y` считаются центром сущности. `width` и `height` используются для прямоугольных коллизий и должны примерно соответствовать видимой части спрайта, а не размеру исходного PNG с отступами.

## Порядок игрового кадра

В `GameScene.update()`:

1. Обновляется фон.
2. Обновляются таймеры run-state, стадии и игрок.
3. По режиму и стадии создаются новые сущности.
4. Вызывается `update()` у сущностей, в том числе у пуль, врагов, boss и pickup-ов.
5. Обрабатываются попадания пуль и взрывов.
6. Обрабатываются столкновения игрока.
7. Удаляются неактивные сущности.
8. Обновляется HUD и shop UI.

Не помещать подбор предметов в `resolveBulletHits()`. Бонусы обрабатываются только в `resolvePlayerCollisions()`.

Если новая механика добавляет взрывной урон, он всё равно должен проходить через общий цикл попаданий и cleanup, а не через отдельные одноразовые обходы массива из UI-кода.

## Подключение новых классов

ES-модули не используются. В конце файла класс экспортируется так:

```js
window.NewEntity = NewEntity;
```

Затем файл добавляется в `index.html` до `GameScene.js`. Порядок `<script>` важен: зависимость должна быть подключена раньше потребителя.

Для конфигов без класса, как в `src/PlayerClasses.js`, глобальный экспорт тоже обязателен:

```js
window.PlayerClasses = { ... };
```

## Работа с ассетами

Новые изображения от пользователя сначала искать в `assets/sprites/source/`.

Предпочтительный формат нового ассета:

- отдельный PNG на один объект;
- прозрачный фон;
- если прозрачность невозможна — строго однотонный `#00ff00`;
- объект целиком в кадре и расположен по центру;
- без текста, рамок, теней и дополнительных предметов;
- имя файла — латиница, kebab-case, без пробелов.

Исходник не менять. Подготовленный вариант сохранять в `assets/sprites/runtime/` под понятным именем. Не подключать случайные UUID-имена напрямую из `source`.

Если у изображения зелёный фон, его можно обработать так:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass `
  -File .\tools\remove-sprite-chroma.ps1 `
  -InputPath .\assets\sprites\source\asset-chroma.png `
  -OutputPath .\assets\sprites\runtime\asset.png
```

После обработки проверить:

- наличие alpha-канала;
- прозрачность углов;
- отсутствие зелёной каймы;
- разумный crop без огромных пустых полей;
- соответствие размера отрисовки коллизиям.

Для новых одиночных ассетов предпочтительнее отдельные PNG. Не расширять общий атлас без необходимости: ручные координаты нарезки легко ломаются.

Все изображения загружаются в `src/Assets.js`. Добавлять загрузку в общий `Promise.all`, чтобы игра не стартовала до готовности ресурсов.

Сейчас проект уже использует отдельные runtime-ассеты для:

- классов `Инженер` и `Медик`;
- стреляющей трески;
- пулемётчика;
- гранатомётчика;
- босса;
- бинта;
- апгрейда двойного выстрела;
- монеты;
- сундука.

Если пользователь положил PNG прямо в `src/`, это не новый стандарт хранения. Такой файл считается исходником, а в игру нужно подключать подготовленную копию из `assets/sprites/runtime/`.

## Игровые сущности и роли

Классы игрока описаны в `src/PlayerClasses.js`:

- `Стрелок` — базовый класс по умолчанию;
- `Инженер` — сейчас уже стреляет ракетами со взрывом;
- `Медик` — пока близок к базовому, но зарезервирован под поддержку.

Текущие типы врагов в `src/Enemy.js`:

- `swimmer` — обычная треска ближнего давления;
- `shooter` — одиночные выстрелы;
- `grunt` — пулемётчик с очередями;
- `bazooka` — гранатомётчик с ракетами и взрывом;
- `BossEnemy` — отдельный босс кампании.

Текущие pickup-ы:

- `HealingPickup` — бинт;
- `UpgradePickup` — постоянный на забег двойной выстрел;
- `CoinPickup` — валюта с разлётом и подбором.

Текущие препятствия:

- `boot`;
- `hook`;
- `chest`.

Сундук разрушается от пуль и рассыпает отдельные монеты, а не один pickup на большую сумму.

## Структура кампании

Кампания больше не должна ощущаться как бесконечный шум. Для этого она собрана из коротких стадий с ограничениями по плотности.

Сейчас до босса используются стадии:

- `Разогрев`;
- `Перекрёстный огонь`;
- `Пулемётная линия`;
- `Подрывники`;
- `Перед Штурмом`.

У каждой стадии свои:

- длительность;
- веса спавна врагов;
- лимит обычных врагов;
- лимиты по типам `shooter`, `grunt`, `bazooka`;
- лимит препятствий;
- шанс сундука;
- разрешение или запрет спавна препятствий.

Если меняется pacing кампании, корректировать нужно именно параметры стадий и caps, а не просто ускорять все глобальные таймеры спавна разом.

## Боссфайт

В конце `campaign` обычный скролл и общий спавн прекращаются, поле зачищается, после чего остаются только игрок и босс.

Текущий босс:

- двигается по вертикали;
- имеет отдельную полоску здоровья в HUD;
- телеграфит веерный залп;
- телеграфит мощный прицельный залп;
- уходит во вторую фазу на низком HP;
- в начале второй фазы выбрасывает один бинт;
- после смерти высыпает много отдельных монет и даёт короткое окно на подбор перед экраном победы.

Telegraph, тяжёлые пули, recoil и смена фазы считаются частью боевой логики. Если меняется паттерн босса, надо проверить не только урон, но и читаемость атаки.

## Текущие параметры баланса

- базовое здоровье игрока: `100 HP`;
- бинт лечит `25 HP` без превышения максимума;
- на полном здоровье бинт не расходуется;
- `Стрелок` и `Медик` стреляют каждые `0.22 секунды`;
- `Инженер` стреляет ракетами примерно каждые `0.42 секунды`;
- апгрейд подбора делает стрельбу двойной до конца текущего забега;
- обычная треска роняет `1` монету;
- стреляющая треска роняет `2` монеты;
- гранатомётчик роняет `3` монеты;
- сундук роняет `5` отдельных монет;
- в кампании есть гарантированный boss encounter, а не бесконечный поток волн.

Баланс хранить рядом с логикой сущности или спавна. Не размазывать одно значение по нескольким файлам.

## Кодировка

Все текстовые файлы должны оставаться в UTF-8. В проекте есть русские строки. PowerShell 5 может неверно отображать UTF-8 без BOM через обычный `Get-Content`; это не всегда означает, что браузер получит битый текст. Для проверки явно указывать `-Encoding UTF8` или проверять байты/страницу в браузере.

Не выполнять массовую перекодировку файлов без проверки результата: так легко заменить русские символы на `�`.

## Проверки перед завершением

Минимальная проверка синтаксиса:

```powershell
Get-ChildItem .\src\*.js | ForEach-Object { node --check $_.FullName }
```

Также вручную или автоматизированно проверить:

- все пути из `Assets.js` существуют;
- новые скрипты подключены в правильном порядке;
- состояние новой механики очищается в `resetWorld()`;
- таймер механики также сбрасывается;
- сущность обновляется, рисуется и удаляется через `cleanup()`;
- здоровье не становится больше `maxHealth` и меньше нуля;
- коллизия соответствует видимому объекту;
- старт, рестарт и game over продолжают работать;
- `campaign` и `endless` оба запускаются с первого экрана;
- boss encounter корректно начинается и завершает run;
- post-run награда и shop не дублируются повторно;
- meta-upgrades реально применяются к следующему забегу;
- pickup-ы собираются только в `resolvePlayerCollisions()`;
- взрывной урон не ломает cleanup и не оставляет неактивные сущности на экране.

## Критерии готовности изменения

Изменение считается законченным, когда оно подключено в реальный игровой цикл, использует runtime-ассет, корректно сбрасывается при рестарте, проходит `node --check` и проверено хотя бы одним целевым сценарием поведения. Просто добавить файл или класс недостаточно.
